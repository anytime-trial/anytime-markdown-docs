# Anytime Markdown 技術アピール戦略

更新日: 2026-03-20


## 1. 現状の技術的強み

| 技術要素 | アピールポイント |
| --- | --- |
| モノレポ構成（4パッケージ） | 設計力・パッケージ分離の判断力 |
| Tiptap/ProseMirror カスタム拡張 | エディタ内部の深い理解 |
| マルチプラットフォーム（Web/VS Code/Android） | 同一コアを3環境に展開するアーキテクチャ |
| Diff/Merge ビュー | アルゴリズム実装力 |
| Mermaid/PlantUML/KaTeX 統合 | 複数ライブラリの統合・制御 |
| Next.js 15 + React 19 | 最新スタック追従 |
| Capacitor によるモバイル対応 | クロスプラットフォーム経験 |
| Playwright E2E + Jest | テスト戦略 |


## 2. やるべきこと


### 2.1 見せ方の整備（最優先）

- **README を「技術紹介」として書き直す** — 機能一覧ではなく、設計判断・技術的チャレンジ・アーキテクチャ図を前面に出す。
- **デモサイト** — Web アプリを Netlify/Vercel にデプロイし、触れる状態にする。\
既にあるなら URL を目立たせる。
- **スクリーンショット / GIF** — Diff ビュー、Mermaid レンダリング、VS Code 上の動作など視覚的に伝わる素材を用意する。


### 2.2 技術記事の発信

記事化しやすいテーマ例:

- 「Tiptap/ProseMirror でカスタムエディタ拡張を作る際の設計判断」
- 「同一エディタコアを VS Code Webview と Next.js で共有する方法」
- 「Markdown エディタに Diff/Merge 機能を実装した話」
- 「Capacitor で Web アプリを Android に展開する際のハマりどころ」

> Zenn / Qiita / dev.to で公開し、リポジトリにリンクを貼る。


### 2.3 コード品質の可視化

- **CI バッジ** — テスト通過率、ビルド状態を README に表示する。
- **カバレッジレポート** — Codecov 等で可視化する。
- **型安全性** — `strict: true` であることを明示する。
- **Lint / Format 統一** — ESLint + Prettier の設定が整っていることを示す。


### 2.4 アーキテクチャドキュメント

リポジトリ内に簡潔なアーキテクチャ図（Mermaid）を置く。\
パッケージ間の依存関係、データフロー、拡張ポイントが一目でわかるもの。\
自分のプロダクトで Mermaid を使っている説得力もある。


### 2.5 コントリビューション実績の見せ方

- コミット履歴が整理されている（Conventional Commits を既に使用中）。
- Issue / PR の運用が見える状態にする。\
1人開発でも Issue → PR → Merge のフローを残す。


## 3. 優先順位

| 順位 | 施策 | 工数 |
| --- | --- | --- |
| 1 | README 書き直し + スクリーンショット追加 | 小 |
